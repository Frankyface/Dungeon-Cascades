# Dungeon Cascades — Decision Log

This is an append-only decision ledger. Each entry records a real decision: what we
chose, why, what we rejected, and the condition that should make us revisit it. Format
per entry:

```
## <date> — <title>
**Chose:** … · **Because:** … · **Rejected:** … · **Revisit if:** …
```

Rules: future sessions **append** new decisions to the bottom; they never rewrite or
delete history. If a past decision is reversed, add a NEW dated entry that supersedes it
and explains why — the old entry stays as the record of what we once believed. The
newest applicable entry wins. Fun-gate and pivot/kill outcomes are recorded here too.

---

## 2026-07-15 — Skill philosophy
**Chose:** Cascade engineering as the primary skill, with execution (drag-path) and build strategy as amplifiers. · **Because:** It fixes the "match-3 feels lucky" problem while keeping a dynamic board that Cam wants. · **Rejected:** A perfect-information / deterministic board (kills the board dynamism Cam wants); a pure execution / twitch game (strategy must matter). · **Revisit if:** The fun gate shows cascades feel random anyway.

## 2026-07-15 — Combat move mechanic
**Chose:** Puzzle & Dragons-style drag-path matching (pick up a tile, drag it, it swaps with each tile it passes, under a move timer). · **Because:** It is the best-proven high-skill-ceiling mechanic in match-3, and it stays turn-based so strategy survives. · **Rejected:** Swap + move timer (lower skill ceiling); fully real-time (drafting stops mattering); timing-window triggers (too light). · **Revisit if:** Drag feels bad with a mouse / on smaller screens, or Skia gesture performance fails.

## 2026-07-15 — Run structure
**Chose:** Slay the Spire-style branching node-map runs (fight / elite / event / shop / rest). · **Because:** It is a proven rogue-lite frame, and route choice adds the build-strategy skill layer. · **Rejected:** Puzzle Quest-style head-to-head; Dungeon Raid-style single continuous board. · **Revisit if:** Playtests show the map layer adds friction without adding meaningful choice.

## 2026-07-15 — Platform
**Chose:** A mobile app (iOS/Android) with iPhone as the primary test device. · **Because:** Drag-path input is touch-native, and Cam tests on an iPhone. · **Rejected:** Web-first; desktop / Steam (a bigger bar, and a later destination if ever). · **Revisit if:** The touch mechanic proves to need a larger screen, or a desktop audience becomes the priority.

## 2026-07-15 — Ambition
**Chose:** A fun project / free game — lean scope, placeholder art, ship early, no monetization. · **Because:** The goal is a game Cam and friends enjoy, not a commercial product; keeping scope lean protects the schedule and keeps focus on the core. · **Rejected:** A polished commercial release; any monetization model. · **Revisit if:** The finished game turns out to have wider appeal and Cam's appetite changes.

## 2026-07-15 — Tech stack
**Chose:** Expo + TypeScript + Skia + Reanimated + Gesture Handler, with a pure headless TS engine, and Jest for tests. · **Because:** It is Cam's known toolchain; all the libraries run in Expo Go (so Stage 1 needs no custom builds); and a headless engine enables sim-based verification. · **Rejected:** Godot 4 (a new toolchain to learn); Phaser-in-Capacitor (drag feel risk in a WebView); Unity (overkill). · **Revisit if:** Skia can't hold 60fps on device.

## 2026-07-15 — v1 scope
**Chose:** IN — a branching node map, shops/events/rest, light meta-unlocks, one starting build, and a boss; run length 15–20 minutes across 8–12 encounters. · **Because:** This is the smallest complete rogue-lite loop that proves the concept. · **Rejected (parked for v2+):** Multiple builds, acts 2+, daily challenges, ascension ladders, cloud saves, monetization. · **Revisit if:** The fun gate passes and appetite changes.

## 2026-07-15 — Theme
**Chose:** Fantasy dungeon-lite (slimes, skeletons, cursed relics). · **Because:** It is instantly readable, free / placeholder assets exist everywhere, and relic/event flavor writes itself. · **Rejected:** Abstract-minimal; sci-fi. · **Revisit if:** The theme starts limiting the mechanics or a stronger hook appears.

## 2026-07-15 — Verification convention
**Chose:** Mostly automated — TDD unit tests on the engine, a sim harness for balance, and Cam plays milestone builds for feel. · **Because:** Cascade logic grows quickly and needs machine-checkable guardrails; only Cam can judge feel. · **Rejected:** A mostly-manual convention (too risky as cascade logic grows). · **Revisit if:** The automated suite stops catching the bugs that actually matter, or maintaining it costs more than it saves.

## 2026-07-15 — Top risk & defense
**Chose:** Treat "the core isn't fun" as the #1 risk, and defend against it by building Stage 1 as a naked board with an explicit kill/pivot fun gate BEFORE any rogue-lite wrapper is built. · **Because:** If the core drag-path cascade loop isn't fun, nothing built on top of it will save it — so prove that first, cheaply. · **Rejected:** Building combat / run / meta systems first and hoping the core lands. · **Revisit if:** The fun gate passes and the core is validated (the defense has done its job).

## 2026-07-15 — Name & repo
**Chose:** Working title "Dungeon Cascades", public GitHub repo Frankyface/Dungeon-Cascades. · **Because:** A working title is enough to start, and a public repo suits a free, for-fun project. · **Rejected:** Holding for a final name before starting. · **Revisit if:** A better final name is chosen (an open question), or the repo needs to change visibility.

## 2026-07-15 — Stage 1 refill RNG model (settled by structured two-agent debate)
**Chose:** Pure uniform random refill for Stage 1, implemented behind a swappable, seeded `TileSource` interface, with a sim-harness flag reserved for comparing alternative sources. · **Because:** In drag-path match-3 the engineered combos are computed from the visible board at release — refill cannot starve them, it only adds bonus skyfall, so droughts don't invalidate the core skill payout (this broke the Tetris 7-bag analogy). Uniform is also the null hypothesis: the sim harness must measure the board's true variance before any smoothing exists, or the stats measure the smoothing instead of the board; and smoothing baked in from day one could mask an unfun core at the fun gate. · **Rejected:** Bag/weighted refill as the Stage 1 default (Tetris-bag fairness argument, fun-gate insurance, tighter sim variance) — deferred, not dismissed; its interface-seam and comparison-flag demands were adopted. · **Revisit if:** Sim stats show skyfall combos rivaling engineered combos for the greedy bot, or the fun gate feels luck-dominated — then build the bag and A/B it by sim flag in Stage 2.

## 2026-07-15 — Build Stages 2–5 ahead of the fun-gate verdict (owner directive)
**Chose:** Proceed with Stage 2–5 implementation now, in parallel with the still-open Stage 1 fun gate. · **Because:** Cam explicitly directed it ("project manage stage 2-5 as you get opus agents to build it") while his TestFlight testing is pending — the owner consciously accepts the risk of building on a not-yet-validated core. · **Rejected:** Blocking all downstream work on the gate (the original plan) — overridden by the owner. · **Revisit if:** The fun-gate verdict comes back pivot or kill — then Stages 2–5 work stops immediately and the verdict governs.

## 2026-07-15 — Tile taxonomy: offense-first affinity model (settled by structured two-agent debate)
**Chose:** Four damage colors + one utility: R/G/B/Y are elemental damage affinities checked against per-enemy weakness/resistance multipliers (weak 2.0 / normal 1.0 / resist 0.5 / rare immune 0.0); P = Heal. ALL cleared groups — heal included — count toward the cascade multiplier, so every combo always pays. Tile effects are data-driven in the engine so new verbs (e.g. Block) can be added cheaply later. · **Because:** "Every combo counts" protects the verified 9.7× skill gap and the "losing was my fault" contract — under resource-mapping, a brilliant cascade of off-role tiles pays nothing and the board (not the player) decides the turn. The genre precedent also favors this shape: P&D itself is five attack elements + hearts, and Slay the Spire keeps block/economy out of the core resource (defense-by-relic is proven). One damage pipeline is also far cheaper to build/balance while the Stage 1 fun gate is still open, and affinity spreads give enemy variety for free. · **Rejected:** Resource-mapped tiles (Attack/Block/Heal/Mana/Gold) — deferred, not dismissed; its best insights were adopted (Heal as the utility color; skill-earned economy returns in Stage 3 as performance-scaled gold rewards; five-hook relic design replaced by affinity-keyed relic hooks). · **Revisit if:** Stage 3 relic design finds too few interesting hooks on affinities, or combat sims/Cam's play find fights one-note — then add a Block tile via the data-driven effect layer and A/B it in sims.

## 2026-07-15 — Combat math defaults (sim-tunable constants, not a design fork)
**Chose:** Per cleared damage group: `ATTACK_BASE × (1 + GROUP_SIZE_BONUS×(size−3)) × affinityMultiplier`. Move damage = Σ groups × cascade multiplier `(1 + CASCADE_BONUS×(totalCombos−1))`, where totalCombos counts ALL groups incl. heals. Heal groups: `HEAL_BASE × (1 + GROUP_SIZE_BONUS×(size−3))`, also cascade-scaled. Starting defaults: ATTACK_BASE 10, HEAL_BASE 5, GROUP_SIZE_BONUS 0.25, CASCADE_BONUS 0.25, player HP 60. All live in one config module. · **Because:** P&D-proven shape; simple, hand-computable for fixtures; every constant is sim-tunable — the combat sim harness is the tuning instrument, with target bands defined in the stage-2 feature specs. · **Rejected:** Debating the curve — it's a tunable, not a fork. · **Revisit if:** Sim win-rate bands (see feature-combat-resolution.md) can't be reached by constant tuning alone.

## 2026-07-15 — Stage 3 structural defaults (adopted, not debated — proven StS conventions; all sim-tunable)
**Chose:** One-act run: seeded layered DAG map ~7 floors deep, 2–3 route choices per floor, node mix ≈ fights majority + ≥1 elite + 1 shop + ≥1 rest + 1–2 events + boss last (8–12 encounters/run). Single-enemy fights in v1. Relics = data-driven passive modifiers with typed trigger hooks (onDamageComputed / onHealComputed / onCombatStart / onGoldEarned…), v1 roster ~12, draft pick-1-of-3 after fights/elites. Economy: performance-scaled gold (base + bonus for fast wins and HP retained — the debate's "skill-earned economy" concession), shop sells relics + a heal, rest heals 30% max HP, ~6 scripted choice events. Difficulty: enemy-stat scalar per floor depth. Player HP persists across nodes. Storage: **async-storage** behind an injected persistence port (engine stays pure); saves on node completion. · **Because:** Every element is the boring, genre-proven default; the interesting design surface (which relics, exact numbers) is data the sims + Cam tune, not architecture. async-storage is Expo Go-compatible and the data is tiny. · **Rejected:** Multi-enemy fights (UI + intent complexity for v1), MMKV (needs a dev build), acts 2+ (explicit v1 non-goal). · **Revisit if:** Full-run sims can't reach the jeopardy bands in feature-run-lifecycle.md by tuning data, or relic hooks prove too thin (taxonomy decision's revisit trigger).

## 2026-07-15 — Meta-progression shape: sim-verified starting variants (settled by structured two-agent debate)
**Chose:** Stage 4 meta-unlocks are **power-neutral starting variants** (sidegrades): vanilla start always available, plus ~6 unlockable alternative starting conditions (e.g. start-relic with −max-HP, +gold with a removed rest site, inverted heal/damage tradeoffs, affinity skews, revealed map with buffed elites). Hard machine gate: each variant's policy-bot win rate over ≥2000 seeded full runs must sit within **±5 percentage points of vanilla** — a variant outside the band cannot ship. Unlock trigger: **cumulative-score tranches** across runs (losses bank progress). · **Because:** The choice at "new run" is the build-strategy skill pillar arriving pre-draft; sidegrades teach deep systems by inverting constraints; and uniquely in this project the full-run sim makes the purity principle machine-checkable per variant — pool-dilution effects are never per-unlock isolatable. At v1 scale, gating an already-small ~12-relic pool would thin runs where they're thinnest. · **Rejected:** Content-pool expansion tranches (StS card-unlock model) as the v1 shape — deferred to v2 when content volume makes gating painless; its cumulative-progress trigger and "identity lives in play" caution were adopted. · **Revisit if:** Content volume grows enough that pool tranches stop thinning runs (then run both shapes), or the variant sim band proves untunable for a proposed variant (drop that variant, not the band).

## 2026-07-15 — Combat recalibration (supersedes "Combat math defaults" constants; its revisit trigger fired)
**Chose:** Recalibrate combat numbers after first sim contact: ATTACK_BASE 10→~3, HEAL_BASE 5→~2 (curve multipliers and PLAYER_MAX_HP 60 unchanged), enemy stats scaled up (direction: slime ~80 HP / attack 8; skeleton ~120 HP / attack 8→charge→16; bat ~90 HP / attack 6 + self-heal 8) — final values are whatever passes the bands, tuned within: ATTACK_BASE 2–5, HEAL_BASE 1–3, enemy HP 40–300, enemy attack 4–20, enemy self-heal 4–12. Combat unit fixtures are recomputed BY HAND from the new decided constants (arithmetic shown in the build report); config-pin assertions now track these decided values as drift guards. **Amended balance bands** (criteria change, flagged for Cam's review): greedy-combat ≥80% win vs each enemy; random ≤40% vs ≥2 of 3; greedy median turns-to-win slime 3–8 (intro enemy), skeleton 5–12, bat 4–12; byte-deterministic reports; Stage 1 board baseline byte-identical. · **Because:** Sim evidence (500 games/pairing, seed 42, default config): greedy realized ~66–76 damage/move vs 30–55 HP enemies → 100% wins at median 1 turn; random over-won vs slime (60.6%) and bat (76.6%) because enemy pressure was too low to punish flailing. An order-of-magnitude mismatch between the damage curve and enemy HP cannot be bridged within the old ±50% bound, and the original uniform 4–12 median floor mis-fit an intentional intro enemy. The tuning agent correctly STOPPED rather than force green — this entry is the authorized correction. · **Rejected:** Enemy HP ~300+ at the old curve (comical numbers, fixtures all break anyway); relaxing bands to fit broken defaults (greedy median 1 = combat without decisions). · **Revisit if:** The new bands still can't be reached within the widened bounds (then the curve SHAPE — e.g. CASCADE_BONUS — is wrong, a real design question), or Cam's on-device play contradicts the sim-tuned feel.

## 2026-07-15 — App identity for store builds (bundle ID)
**Chose:** iOS bundle identifier + Android package `com.frankyface.dungeoncascades`; display name "Dungeon Cascades", slug `dungeon-cascades`, URL scheme `dungeoncascades`. · **Because:** Cam delegated the bundle-ID choice ("make your own"); reverse-DNS on his GitHub handle (Frankyface) is a stable namespace, all-lowercase-alphanumeric so it is valid for both stores. Fixed the leftover `tmp-expo` name/slug/scheme from scaffolding at the same time. · **Rejected:** A throwaway `com.example.*` id — bundle ids are effectively permanent once an App Store Connect record exists, so the real one is chosen now. · **Revisit if:** Cam picks a different final game name — the display name can change freely, but the bundle id should NOT change after the App Store Connect record is created.

## 2026-07-15 — Stage 1 board defaults adopted
**Chose:** 6×5 board (6 columns × 5 rows), 5 tile colors, ~5s drag timer enforced in the UI layer (engine stays timer-agnostic), orthogonal-only drag. · **Because:** These were the documented proposed defaults (P&D-standard board); adopting them unblocks the engine build, and all are cheap constants to change. · **Rejected:** Debating each default individually before writing code. · **Revisit if:** The fun gate or sim stats argue for a different board size, timer, or diagonal drag.
